C/C++ regex extractor no longer captures `<MODULE>_API` export macros as class names (#1693 v0.89). Second fix from the #1389 cross-language false-positive sweep. C++ codebases using a dllexport macro — `class PYRITE_API AMyActor`, the Unreal Engine `<MODULE>_API` convention — had `classRE` capture the macro token (`PYRITE_API`) as the class name instead of the real type. Every `class <MODULE>_API X` in a file then collided on the macro name (the sweep saw `PYRITE_API appears 6 times` in one header) and `pincher doctor` reported `qualified_name_collision`. `classRE` now skips an optional leading ALLCAPS-with-underscore export-macro token: `(?:class|struct|union)\s+(?:[A-Z][A-Z0-9_]*_[A-Z0-9_]*\s+)?(?P<name>…)`. The macro group requires an underscore (so single-word ALLCAPS class names aren't eaten) and is optional + greedy — when there's only one identifier after `class`, RE2 takes the not-taken branch and captures it directly, so an ALLCAPS-underscore class name with no second identifier (`class API_REGISTRY`) is still kept. Base-class clauses (`class PYRITE_API AThing : public AActor`) parse correctly with the macro skipped.
