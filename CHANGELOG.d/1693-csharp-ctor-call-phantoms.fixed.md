C# regex extractor no longer emits phantom Method symbols for object-creation and brace-less control-flow statements (#1693 v0.89). Fifth fix from the #1389 cross-language false-positive sweep. funcRE's `(returntype)+ name (` form treats a leading statement keyword as a return-type token, so `new Vector2(...)`, `return new Foo(...)`, the object-initializer `new PrototypePlacedObjectData() { ... }` (which recurs 195× in one corpus file), and brace-less control arms like `else GetTree().Foo()` / `do Tick()` each emitted a phantom Method named for the constructed type or the called method — colliding on qualified_name with the real definition. New `dropCSharpConstructorCalls` filter drops a Function/Method whose declaration line opens (after indent) with a statement keyword (`return` / `throw` / `await` / `else` / `do` / `yield return`) or with `new <Name>(`. The discriminator is precise: a real method declaration never has `new` immediately before its name (the `new` method-hiding modifier is always followed by the return type), so `new <Name>(` is unambiguously object creation while `new <ReturnType> <Name>(` stays a real method. Reindex impact: Yevolve 9→1, Pyrite-Booty 5→1 qualified_name_collision rows.
