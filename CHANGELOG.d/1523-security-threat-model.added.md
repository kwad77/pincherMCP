**Security threat model documentation ([#1523 FILE-D](https://github.com/kwad77/pincher/issues/1523) v0.87):** new `docs/security/threat-model.md` enumerates the six attacker surfaces pincher exposes (HTTP gateway, `fetch` tool, cypher query injection, path traversal in `init`/`index`/`fetch`, credential exposure in stats/health/doctor, watcher + indexer file consumption), the current mitigation status of each, and what's deferred to v1.x or out of scope. Pre-doc, every security-adjacent fix (#844 DNS-rebinding, #607 credential redaction, #41 symlink-safety audit, bloat-trap refusal) lived in isolation — a reviewer asking "what's pincher's security posture?" had no consolidated answer. The doc declares the trust boundary explicitly (developer who launched pincher is trusted; the LLM agent calling MCP tools is not; the contents of indexed files are data, never executed), enumerates per-surface mitigations + known gaps + explicit deferrals, and documents what's explicitly NOT pincher's threat model (LLM prompt injection lives at the host layer; malicious source files crashing the indexer is a quality bug not a CVE; users indexing directories containing secrets is expected behavior, don't do that). The reachability section pairs with the FILE-F (#1525) govulncheck flow: `-show traces` evidence determines whether a transitive-dep CVE is a release blocker or a SECURITY.md exception entry. Forms the v0.87-v0.88 supply-chain hardening triplet alongside FILE-E (#1524 artifact signing) and FILE-F (#1525 govulncheck CI gate). Re-audited at v0.84 API freeze and v0.90 stable promotion per the doc's stated cadence.
