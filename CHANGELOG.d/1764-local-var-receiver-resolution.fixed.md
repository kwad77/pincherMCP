Go method calls through a local-variable receiver now resolve their CALLS edge (#1764 v0.91, #1747 sibling). A method reached only via `w := &Worker{}; w.Run()` dropped its edge — `extractGoCalls` stamped `ReceiverType` from the enclosing method's receiver, and #1747's pre-pass covered only package-level vars, so a local-var receiver carried no hint and the #423 `resolveByReceiverType` binding had nothing to bind. A method only ever reached that way looked dead to `dead_code` / `audit_unused`. The Go extractor now runs a function-body local-var type pre-pass (`x := &T{}` / `x := T{}` / `x := new(T)` / `var x T` / `var x = &T{}` — the shapes whose type is unambiguous), stamping the local's type on `x.method()` calls; a local-var receiver shadows a package var of the same name and wins. `resolveByReceiverType` case-2 additionally tries both the pointer and value receiver forms, so a value-typed local (`w := Worker{}`) still binds a pointer-receiver method (Go auto-addresses). Constructor-call inference (`x := NewT()`) needs cross-symbol return-type lookup and stays a follow-on. Found in the v0.91 issue-log triage.
