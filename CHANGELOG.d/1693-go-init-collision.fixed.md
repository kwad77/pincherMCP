`qualified_name_collision` diagnostic no longer false-fires on Go's multiple `func init()` (#1693 v0.89). Third fix from the #1389 cross-language false-positive sweep. Go permits multiple `func init()` per file — they all run, in source order, by language design. The extractor correctly emits one symbol per `init()` and `disambiguateDuplicates` suffixes them `~<line>` so every one survives; but `recordExtractionHeuristics` then scored the `init` QN as a collision and recorded a `qualified_name_collision` extraction failure. Every Go file with 2+ `init()`s tripped a false positive — in pincher's flagship, AST-tier language. The diagnostic now skips a colliding QN whose final `.`-segment is exactly `init` when the language is Go (new `qnLastSegmentIsInit` helper); when that's the *only* colliding QN, no failure row is recorded at all. Narrow carve-out — a Go file with a genuine non-`init` duplicate-QN collision is still diagnosed, and `pkg.initialize` (substring, not segment) is unaffected.
