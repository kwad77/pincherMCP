**list (#1473 v0.78):** `_meta.counts_may_be_transient: true` + warning when `prune_dead=true` actually deleted projects. The prune CASCADE deletes files/symbols/edges rows whose paths live under pruned project paths; where files belong to multiple projects (parent project + worktree sub-project sharing physical paths), the cascade temporarily reduces the JOIN counts the SELECT reports for surviving projects. A user reading the response payload pre-fix would reasonably think the prune nuked their indexes (counts collapsed 80-95% mid-cascade, recovered within ~60s after the watcher reindexed). Now the response flags the transient state: typed boolean for hosts that branch on `_meta` shape (per the meta-envelope contract); human-readable warning string for hosts that surface `_meta.warnings` in their UI. Only fires when the prune list is non-empty (no-op prunes can't trigger the transient state). Option (c) from the issue — the simpler alternative (a) "skip the projects array entirely when prune_dead=true" would have been a contract change; (b) "run the SELECT AFTER a watcher recovery cycle synchronously" needs deeper indexer coordination. The flag-and-warn shape preserves the existing response contract.
