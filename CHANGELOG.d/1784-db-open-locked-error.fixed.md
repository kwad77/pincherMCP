CLI commands now print an actionable message instead of a raw `SQLITE_BUSY` when the database is write-locked (#1784 v0.89). `db.Open`'s `migrate()` step takes a write lock; when another pincher process holds the writer past the 5s `busy_timeout` — a force-reindex of a large repo easily does — the open failed with `pincher: failed to open database: migrate: init schema_version: database is locked (5) (SQLITE_BUSY)`. That hit `pincher doctor` in particular — the command users run *when pincher seems stuck* is exactly when the DB is most likely write-locked. `db.Open` now detects the lock contention and returns `database is locked — another pincher process is writing it (an index is likely in progress); retry in a few seconds`, with the raw cause still wrapped for debugging. The fix is at the shared open path, so every CLI subcommand (`doctor`, `stats`, `rebuild-fts`, `verify`, ...) inherits it. Found in the v0.89 workflow-accuracy deep dive.
