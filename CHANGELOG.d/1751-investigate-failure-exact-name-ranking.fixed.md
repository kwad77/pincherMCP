`investigate_failure` now ranks the functions actually named in a stack trace, not unrelated methods of the same receiver type (#1751 v0.89). A Go panic naming `(*Indexer).resolveCalls` / `(*Indexer).Index` ranked neither — the top suspects were `New` / `anyActive` / `flushBatch` and other `*Indexer` methods that merely shared the receiver-type token `Indexer`. Two compounding bugs: (1) the Step-2 search loop broke once 15 candidates accumulated, and `frameNames` is ranked by raw token frequency — a Go trace repeats package-path components (`internal`, `index`, `github.com`, `com`) once per frame, so they out-ranked the once-mentioned function names and the loop exhausted its budget before `resolveCalls` / `Index` were ever searched; (2) every frame match scored a flat `+0.45`, so a loose BM25 hit (the token `Indexer` appearing in `New`'s return type) tied an exact hit, and `caller_fan_in` then let the hub function `New` edge out the real crash site. The loop now searches every parsed frame name (bounded by name count, not candidate count), and an exact-name frame match (`symbol.Name == frameToken`) scores `+0.45` with a `stack_frame_exact_name_match` evidence flag while a loose match scores `+0.20`. Found dogfooding the composite-tool surface.
