Go resolver: method calls through package-level singleton vars now resolve (#1747 v0.89). A type only ever instantiated as a package-global singleton (`var pyRE = &regexExtractor{}`) had every method on it look 100% dead — `extractGoCalls` stamped `ExtractedEdge.ReceiverType` with only the *enclosing* method's receiver type, so a call `pyRE.extract(...)` from a free function carried no receiver-type hint and the #423 `resolveByReceiverType` binding had nothing to resolve against. The edge dropped, and `dead_code` / `audit_unused` confidently flagged `*regexExtractor.extract` (15+ call sites — the core regex-extraction dispatch method) as `confidence: high` / `deep_trace_confirms_unreachable` / safe-to-delete. The Go extractor now pre-passes the file collecting package-level `var X = &T{}` / `T{}` / `new(T)` / `var X *T` declarations to a same-package type map; a 2-segment `IDENT.method` callee whose `IDENT` is a known package var (and not shadowed by a local / param / receiver) is stamped with the var's type, which `resolveByReceiverType` case-2 binds. Same-package type idents only — qualified types stay unresolved (the resolver can't map foreign packages without import-graph awareness). Local-var receivers (`x := &T{}; x.M()`) are a sibling gap tracked separately. Found dogfooding `audit_unused` against pincher-repo.
