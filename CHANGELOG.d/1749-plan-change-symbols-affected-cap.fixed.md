`plan_change` no longer emits an unbounded `target.symbols_affected` list (#1749 v0.89). #1727 capped `plan_change`'s `blast_radius` lists but missed `target.symbols_affected` — the file-path resolution branch enumerated *every* callable symbol in the file (`internal/server/server.go` has hundreds), so a `plan_change` against a large file produced a 90 KB response the host could not receive, and Step 2's per-symbol inbound trace loop ran tens of seconds (64s on `internal/ast/extractor.go`). `symbols_affected` is now capped at a fixed 50 (`maxAffectedSymbols`), with `target.symbols_affected_total` keeping the true count and `target.symbols_affected_truncated` flagging the cut — same pattern as #1727's `blast_radius.truncated`. The cap also bounds the trace loop, fixing the latency. The advisory `related_adrs` list — a secondary contributor when a file's package matched dozens of session-log ADR bodies — is capped at 20. No tool-contract parameters added; the cap is fixed, keeping the v1.0 input surface frozen. Found dogfooding the composite-tool surface.
