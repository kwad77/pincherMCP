**pinchQL string literals weren't unescaped, so exact-match `WHERE` on a Windows file path silently matched nothing.** The tokenizer's scan loop already treated `\` as an escape — it skips the char after `\` so `\"` doesn't terminate the literal early — but the token value was the raw substring, backslashes and all. A literal like `"D:\\ClaudeCode\\proj"` compared as the double-backslash string against the single-backslash stored value and never matched, so `WHERE n.project_id = ...` / `n.file_path = ...` / `n.id = ...` returned zero rows on every Windows-indexed project (`CONTAINS` worked only when the needle happened to be backslash-free). The new `unescapeString` completes the escape the scanner already committed to: `\\` → `\`, `\"` → `"`, `\'` → `'`, `\n`/`\t`/`\r` → control chars; an unrecognised escape (`\d` in a regex literal) keeps the backslash verbatim so regex predicates still work. Found during v0.56 dogfooding.
