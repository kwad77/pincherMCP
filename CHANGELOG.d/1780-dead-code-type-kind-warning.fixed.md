`dead_code` and `audit_unused` now warn when `kinds` is widened to a type-shaped kind (#1780 v0.89). The graph tracks `CALLS`/`READS`/`WRITES`/`IMPORTS` edges but no type-reference edge, so a struct used only as `&T{}` carries zero inbound edges and looks dead — on pincher-repo, `dead_code kinds=Function,Method,Type,Class` returned 38/40 false positives (`regexExtractor`, every language extractor, `pythonRunner`, ...). Both tools' `kinds` descriptions previously suggested `Class` as an unqualified widening. Now: requesting `Class`/`Type`/`Interface`/`Enum` stamps a `_meta` warning (`warnings` for `dead_code`, `warnings_v2` code `type_kinds_no_reference_edges` for `audit_unused`) flagging type-kind results as high-false-positive and advisory-only — not safe-to-delete verdicts; the param descriptions and the `audit_unused` empty-result diagnosis carry the same caveat. Found in the v0.89 workflow-accuracy deep dive.
