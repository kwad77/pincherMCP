TypeScript receiver-type resolver lands — closes the v0.61 TS receiver-type stack. The TS extractor now tracks variable→type bindings within function/method bodies (`const x: Cart`, `function f(c: Cart)`, parameter type annotations) and stamps `ReceiverType` on CALLS edges for `this.X()` and `varname.X()` shapes. The resolver then binds these precisely to `Class.method` via a class-name lookup (TS's multi-segment module paths require a different shape than Go's pkg-prefix path). Headline win: when two classes both define `add()`, `(c: Cart).add()` binds to `Cart.add` only — pre-fix the name-only Method fallback could pick either, the canonical dead_code FP family for TS code. Foundation from #1158 (v0.61 per-file CALLS pass + Method symbol extraction) carries forward unchanged. Bottom/top TS types (`void`, `any`, `unknown`, `never`, `null`, `undefined`) drop from the type map so the resolver doesn't chase phantom Classes. New extractor tests pin every shape (this-method, typed-param, typed-local, untyped-control, bottom-type-drop, regex-tier confidence parity); new resolver tests gate the two-classes-with-same-method-name acceptance case.
