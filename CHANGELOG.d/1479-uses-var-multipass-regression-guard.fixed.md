USES_VAR multi-pass persistence regression guard (#1479 v0.89). #1479 (filed against v0.73) reported Ansible USES_VAR edges showing in `pincher index --json-summary` but absent from the DB — a second index pass (the watcher, hash-skip-clean) wiped them via resolveUsesVar's DELETE-then-INSERT when its in-memory pending list was empty. The bug is fixed on current code by two intervening changes: #1627 routes USES_VAR pending edges through `pending_edges` persistence (`appendDeferred(deferredUsesVar)`), so the resolver always reloads them even when the in-memory list is empty; and #1629's incremental resolve skips the resolve block entirely on a true no-change pass (totalFiles==0) and scopes the resolve to changed files otherwise, so an unrelated-file edit can't wipe USES_VAR edges from untouched files. The existing `PersistsAcrossForceReindex` test covered the force-reindex angle; this adds `TestIndex_AnsibleUsesVar_SurvivesNoChangeReindex_1479` (the true no-change second pass — the exact watcher scenario from the bug report) and `TestIndex_AnsibleUsesVar_SurvivesUnrelatedFileEdit_1479` (a changed-but-unrelated-file pass exercising the #1629 scoped resolve). Both pass — the v0.73 wipe cannot recur.
