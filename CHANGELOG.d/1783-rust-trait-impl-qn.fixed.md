Rust methods in separate `impl` blocks on the same type no longer collide on `qualified_name` (#1783 v0.89). `impl Debug for Foo` and `impl Display for Foo` both scoped their `fmt` method to `Foo`, producing two `Foo::fmt` symbols тАФ a real Rust corpus showed 359 such `qualified_name_collision` extraction failures, each silently dropping one of the colliding methods so `trace` / `dead_code` / `neighborhood` missed it. The regex-extractor framework now tracks the QN scope (`currentClassQN`) separately from the Parent scope: a Rust trait-impl QNs its methods as `Type::Trait::method` (`Foo::Debug::fmt` vs `Foo::Display::fmt`) while the `Parent` field stays the bare `Type`. Inherent `impl Type { ... }` methods and every non-Rust scope kind (class / interface / enum) are unchanged тАФ `currentClassQN` equals the scope name there. Found in the v0.89 deep-dive `doctor` extraction-failures audit.
