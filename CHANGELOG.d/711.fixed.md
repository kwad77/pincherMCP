**`make build` survives a locked `pincher.exe` on Windows ([#710](https://github.com/kwad77/pincher/issues/710), follow-up to [#705](https://github.com/kwad77/pincher/issues/705)).** #708 fixed the destination-side file lock during the swap step (rename-out trick on `C:\tools\pincher.exe`), but the source-side `go build -o pincher.exe ./cmd/pinch/` still hit `open pincher.exe: The process cannot access the file because it is being used by another process` if anything was holding the project's `pincher.exe` open — `pincher web` or a direct `--http` launched from the project dir is the common case. New `scripts/build.sh` always builds to `pincher{.exe}.new`, then atomically renames over the target (Windows allows renaming over a locked file because handle resolution is by inode, not path). The Makefile `build:` target now delegates to this script so `make install` end-to-end works even when the dev was probing the dashboard locally moments earlier. Caught minutes after #708 shipped by the next dogfood iteration — exactly the kind of follow-up the user's "point-iterate before tag" mandate exists for.
