**fix(ast/bash):** Bash extractor emits a per-file Module symbol so top-level CALLS edges can resolve at the indexer's per-file `nameToID` stage ([#1482](https://github.com/kwad77/pincher/issues/1482)). Pre-fix `bashEdgeFromName` returned `""` for top-level invocations (`worker  # bare call at file scope`), with a code comment claiming "the indexer treats that as the file-scope edge attachment point" — but the indexer has no such handling. The empty `FromQN` failed `nameToID[""]` lookup, the `parts[0]==""` fallback also failed, and the Bash-doesn't-defer fallback at `indexer.go:706` dropped the edge silently because `lang != Go|Python` and `ReceiverType == ""`. Result: 100% of top-level CALLS lost in production. Post-fix bash emits a per-file Module symbol (matches Go's per-file Module convention at `extractor.go:565`) with `QualifiedName = filename-without-ext`, and `bashEdgeFromName` returns that module name (instead of `""`) for top-level scope. The indexer's `nameToID[module]` now resolves to the Module symbol's ID and the edge persists. Pinned by `TestIndex_BashIntraFileCalls_PersistedToDB_1482` (positive — intra-function calls) + `TestIndex_BashTopLevelCallsToLocalFunc_PersistedToDB_1482` (production-failure-shape — top-level calls to a local helper, the pattern `pincher-repo/scripts/*_test.sh` files use heavily). #1481 (Markdown REFERENCES emit zero edges) and #1479 (Ansible USES_VAR extracted-but-not-persisted) tracked separately — likely same drift class but different root causes.
