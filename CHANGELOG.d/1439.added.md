pinchQL supports `WHERE col IN [a, b, c]` — the canonical "any of these values" list-membership predicate (`WHERE n.kind IN ["Function", "Method"]`). Pre-fix the open-since-v0.8.0 friction was a verbose OR-chain workaround. `NOT IN` works via the existing `NOT` wrapper. Empty list (`IN []`) and paren-list (`IN (...)`) both surface specific errors rather than silent zero-rows. SQL pushdown is parameterised so SQLite affinity coerces literals to column type (numeric IN against INTEGER works) (#1439).
