Cross-file package-var singleton method calls now resolve their CALLS edge (#1778 v0.89, #1747 follow-up). A Go method call through a package-level singleton (`defaultRunner.execute(...)` where `var defaultRunner = &runner{}` is declared in a sibling file of the same package) dropped its CALLS edge: #1747's extract-time singleton pre-pass is file-scoped, so the call carried no `ReceiverType` and the #423 receiver-type binding had nothing to resolve against. The visible symptom was `dead_code` / `audit_unused` confidently flagging live hot-path methods (e.g. `(*pythonRunner).extract`, the #1626 Python daemon entry point, called from `python_ast.go`) as `high`-confidence "safe to delete." The extractor now stamps the inferred singleton type (`&T{}` / `T{}` / `new(T)` / explicit `var X *T`) into the Variable symbol's signature, and `resolveCalls` recovers it cross-file to complete the binding. Found in the v0.89 workflow-accuracy deep dive.
