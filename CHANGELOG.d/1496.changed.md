**Watcher reindexes projects serially instead of fanning out per-project goroutines ([#1496](https://github.com/kwad77/pincher/issues/1496), #1474 follow-up):** pre-fix `Watch()` launched one goroutine per changed project per tick, which on a multi-project setup grabbed every project's cross-process lockfile simultaneously after a schema-migration auto-restart. User `force=true` calls during that window collided with held lockfiles across the board (the #1474 repro: 6 projects, 4 of 6 user force calls fail with "already being indexed"). Post-fix `Watch()` calls `Index()` synchronously per project so at most one project's lockfile is held at any time — a user `force=true` against any idle project wins the lockfile race immediately. Total wall-clock to converge across all changed projects is unchanged (SQLite is single-writer at the WAL anyway, so the goroutines were not actually running in parallel; the only thing the fan-out bought was simultaneous lockfile holding). Pinned by `TestWatch_SerializesPerProjectIndex` (positive — instruments via `SetEventHook` to assert max-concurrent `Index()` from the watcher == 1 with 2 changed projects) + `TestWatch_SingleProjectStillReindexes` (control — single-project case still reindexes after the fan-out removal).
