The `symbols` batch tool is now session-scoped by default, no longer leaking cross-project rows (#1799 v0.91). A symbol ID is `path::qualified_name#kind` — not project-qualified — so an indexed mirror of the session repo carries byte-identical IDs. `handleSymbols` only scoped its lookup when an explicit `project` arg was passed; with `project` omitted (the common case) it ran `GetSymbolsByIDs("", ids)` — an unscoped lookup that resolved each ID from whichever project's row it found first. With a mirror indexed, `symbols ids=["internal/db/db.go::db.Open#Function"]` against session project `pincher-repo` returned the *mirror's* `db.Open` (wrong source / line numbers); the handler detected the leak post-hoc and warned, but had already returned the wrong data. The single-symbol `symbol` tool and `context` / `trace` / `neighborhood` all carry the #1232 / #1408 session-scoping fix — the batch `symbols` was the one variant missed. It now defaults to the session project; an ID genuinely absent from it surfaces as `not_found`; a new `cross_project=true` opts into the unscoped fallback for the missed IDs (a symbol resolved from another project still warns). Found in the v0.91 dogfood loop probing the batch tool surface.
