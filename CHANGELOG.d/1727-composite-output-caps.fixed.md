`plan_change` and `investigate_failure` no longer emit unbounded responses that exceed the MCP per-call token cap (#1727 / #1723 v0.89). Both composites traced inbound callers and serialised every result: `plan_change` against a hotspot-heavy file (an indexer holding `Index`/`New`) produced a 464 KB response, and `investigate_failure` on a stack trace naming a hub symbol produced 270 KB — in both cases the host cannot receive the result at all. Each now caps its caller lists at a fixed 50 rows (matching `trace`'s `max_hops` and `architecture`'s hotspot cap): `plan_change`'s `blast_radius.summary.*_count` keep the true totals with a new `blast_radius.truncated` flag, and `investigate_failure` adds `callers_total` + `callers_truncated` alongside the capped `callers` list. No tool-contract parameters added — the cap is fixed, keeping the v1.0 input surface frozen. Found dogfooding the composite-tool surface.
