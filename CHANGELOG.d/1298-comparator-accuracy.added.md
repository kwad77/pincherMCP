Added an accuracy dimension to the code-intelligence comparator harness (#1298 §2 v0.89), completing the "reports tokens / accuracy / latency" acceptance. `scripts/comparator-raw-readgrep.sh` now scores each comparator on every task with a `*_accurate` boolean — did the tool's output actually contain the right answer? — checked against a grep-based oracle derived independently of any one tool (the symbol's defining file for find-symbol, the declaration line for read-with-context, a real caller file for find-callers), so no comparator is scored against its own output shape. Tokens + latency say how cheap a result was; accuracy says whether it was a result at all. Output JSON bumps to `schema_version: 4` with `pincher_accurate` / `comparator_accurate` / `ctags_accurate` per task (the ctags flag is null on find-callers, which a definition index structurally cannot answer). The accuracy work surfaced and fixed a latent bug in the find-callers task: the pinchQL query selected `caller.file` — not a valid symbol property, so the column came back empty — corrected to `caller.file_path`.
