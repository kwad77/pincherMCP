`query` with `ORDER BY` now sorts the full match set instead of an arbitrary sample. The node-scan path applied the `max_rows` safety cap as a SQL `LIMIT` with no `ORDER BY` in the `SELECT`, then sorted the truncated rows in Go — so `ORDER BY complexity DESC LIMIT 3` returned the top 3 of a random `max_rows`-sized slice, not the global top 3. On the pincher-repo self-index (~2200 functions, default `max_rows` 200) the three genuinely most-complex functions never appeared in the result. `ORDER BY` on a bare `var.property` is now pushed into the SQL `SELECT` so the database sorts the full set before the cap truncates it; aggregate `ORDER BY` (e.g. `COUNT(n)`) is unaffected (still handled post-scan). Tests missed this because fixture datasets are smaller than `max_rows`, making cap-before-sort and cap-after-sort identical (#847).
