`context_for_task` task-mode now seeds on production code before test functions (#1776 v0.89). The composite's task-driven path BM25-searched the code corpus and took the top callable hits as seeds with no test-vs-production preference. Test functions carry BM25-rich names (`TestExecute_UnknownPropertyDedupAcrossClauses`) that out-rank the terse implementation (`Execute`) for a "how does X work" query — so `context_for_task task="how does the cypher engine reject multiple MATCH clauses"` seeded on two test functions and an interface, left the actual implementation as a depth-2 callee, and flooded `neighbors` with ~45 sibling test functions from the seed's `_test.go` file. Seed selection is now two-pass: production (`!is_test`) callable hits fill the seed slots first; test symbols only fill any remainder — so a task that genuinely targets a test ("fix failing TestFoo") still seeds, but the dominant "how does X work" / "fix bug in X" intent gets the implementation.
