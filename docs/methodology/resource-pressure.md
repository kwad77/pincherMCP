# Resource pressure — peak RSS by file count

Per-tier peak resident set size measured against synthetic Go corpora. Anchors the **"minimum recommended RAM per indexed-file tier"** budget published in `docs/reference/architecture.md` and gates regression at v0.90+ per FILE-I (#1528).

## What it measures

`scripts/resource-pressure-bench.sh` runs `pincher index` against four file-count tiers (1k / 10k / 50k / 100k) under `/usr/bin/time -v` (Linux) or `/usr/bin/time -l` (macOS). The "Maximum resident set size" line captures the indexer process's peak RAM use, including the SQLite WAL pages that bloat under heavy single-writer churn.

Each tier is a fresh synthetic corpus generated by `scripts/generate-synthetic-corpus.sh`:
- ~1 KB of valid-but-trivial Go per file.
- 100-bucket directory splay so `gocodewalker`'s readdir isn't the bottleneck.
- 1 package decl + 1 import + 1 exported function + 1 exported variable per file — enough surface for the extractor to do real work.
- A real `go.mod` so the indexer treats the corpus as one project.

## CI coverage

`.github/workflows/resource-pressure.yml` runs weekly (Tuesday 13:00 UTC) + on `workflow_dispatch`. Default tiers in CI are 1k + 10k — 50k and 100k are available via dispatch with `TIERS=...` but disk + runner-time cost makes them not-on-cron. Trigger explicitly when validating the published budget for the larger tiers.

Output: JSON per run, uploaded as the `resource-pressure-<run_id>` artifact (90-day retention).

## Phase rollout

| Release | Workflow `continue-on-error` | Effect |
|---|---|---|
| v0.85-v0.89 | `true` (advisory) | Numbers feed the `docs/reference/architecture.md` budget rows. Regressions log `::warning::` but don't block. |
| v0.90+ | `false` (hard budget) per FILE-I | Peak RSS exceeding the published per-tier max blocks the release tag. |

## Why Linux-first

Windows `pincher index` is dominated by NTFS overhead — a Linux RSS number tells you nothing about how the indexer behaves on Windows. We publish a Linux/macOS-only budget rather than a misleading cross-platform claim.

macOS support is scaffolded (the `time -l` parser exists) but not run in CI; macOS reporting is for `workflow_dispatch` use only.

Tracking Windows separately is filed as a v1.x task (would need pagefile + commit-charge instrumentation, not RSS).

## Published budget shape (reference docs)

Once a few weeks of phase-1 numbers settle, the `docs/reference/architecture.md` "Resource requirements" section grows a table:

```
| Indexed files | Recommended RAM | Peak RSS (CI baseline) |
|---|---|---|
| ≤1k    | 256 MiB | ~XXX MiB |
| ≤10k   | 512 MiB | ~XXX MiB |
| ≤50k   | 1 GiB   | ~XXX MiB |
| ≤100k+ | 2 GiB   | ~XXX MiB |
```

Numbers in the "Peak RSS" column come from this bench; the "Recommended RAM" column is double-the-measured-peak rounded to the next stable RAM tier (gives headroom for the OS + other processes the user is running).

## Out of scope (FILE-I follow-ups)

The original FILE-I acceptance has four bullets; this PR ships the first two:

1. ✅ Bench harness + synthetic-corpus generator.
2. ✅ Published budget rows (placeholders; numbers fill in over phase-1).
3. **Graceful degradation**: when memory is tight, watcher polls less aggressively (configurable). Deferred — needs runtime memory-pressure detection in the watcher loop. Filed as a v0.86 follow-up.
4. **Failure-mode test**: indexer aborts cleanly on OOM (no half-written DB). Deferred — needs WAL-level mid-flush abort handling. Filed as a v0.87 follow-up.

The bench-only scope lets us PUBLISH the budget at v0.85 (real numbers) and ENFORCE it at v0.90 (regression gate). The runtime-degradation work is the harder lift and benefits from having the bench in place first.
