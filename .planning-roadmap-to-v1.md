# Roadmap to v1.0 — Replanned

**Status:** Draft for review. Replaces the per-release theme tables in
`#669` (Phase 4) and `#667` (Phase 5) which decoupled from reality
once Phase 4 composites started shipping (v0.81 was "related-repos
prototype" but is shipping `investigate_failure`; v0.82 was "polyglot
stress" but is shipping `plan_change`). This doc is the post-correction
plan.

**Constraints baked in:**

1. **One composite per release wastes ramp.** Each minor packs 3–5
   substantial planned items + a reserved dogfood window. No more
   "one feature ships."
2. **.x9 = hardening, .x0 = stable retag.** Per memory + #672 umbrella.
   v0.89, v0.99 don't take features. v0.80, v0.90, v1.0 are channel
   re-tags of the prior `.x9` binary.
3. **31 scheduled issues + ~20 newly-identified gaps need slots.** Math
   below.
4. **Hardcore dogfood is structural, not opportunistic.** Every minor
   reserves explicit slots; every release ships a CHANGELOG entry
   summarizing friction-found-and-fixed.
5. **Composite contract from `docs/integrations/composite-tool-roadmap.md`
   still holds.** Additive, no internal MCP round-trips, single
   envelope, empty_reason mandatory, idempotent.

## Hardcore dogfood — structural changes

Per request, dogfood is structural not opportunistic. Concrete
mechanisms going into every release:

1. **Dogfood reserve slots are budgeted, not aspirational.** The
   per-minor allocations above name explicit reserve counts. If a
   release doesn't fill its reserve with friction-found PRs, that's
   a signal to *probe harder*, not to ship early.
2. **Every release ships a "DOGFOOD:" CHANGELOG section** —
   bullet-listing friction found AND fixed in that window. Friction
   found but NOT fixed gets filed as an issue with `dogfood-found`
   label, not silently dropped.
3. **Probe rotation per session** — every dogfood session probes a
   different axis: tool surface, language extractor, error path,
   integration host, CLI subcommand, doctor advisory. The memory
   note `feedback_dogfood_pincher_aggressively.md` already names
   this; making it structural means tracking which axis got probed
   in which release.
4. **Friction-found issues route by severity, not by milestone.**
   Severity-1 (broken canonical workflow) → next patch release.
   Severity-2 (silent-wrong / misleading output) → next minor.
   Severity-3 (polish / ergonomics) → next dogfood reserve slot.
5. **"Filed-and-shipped" sessions are the gold standard.** Per memory
   (v0.72 session2: 8 of 11 PRs were file-and-fix in the same slice).
   Each release should target ≥3 such cycles.
6. **External-corpus dogfood at v0.85, v0.91, v0.95.** Not internal
   pincher-repo only; index an unfamiliar repo (Linux kernel slice,
   a popular OSS project) and run the canonical workflows. Anything
   that breaks gets filed.

## Dogfood routing — what to do with net-new work that arrives mid-flight

The reserve slots above budget *capacity*. This section is the
*routing rule* for what type of net-new discovery lands where, so
dogfood-found work doesn't silently bloat planned scope or get lost.

### Routing by type, not just by severity

| Discovery shape | Lands in |
|---|---|
| **Regression in shipped code** — canonical workflow broken | Next patch (v0.81.x / v0.82.x). Ships within days, doesn't wait for the next minor. |
| **Bug in current in-flight PR** | Same PR or a sibling PR before merge. Never punted forward. |
| **Bug in shipped feature, silent-wrong / misleading** | Next minor's dogfood reserve slot. |
| **Net-new capability gap** — missing composite / doctor advisory / tool | Triage: v1.0 blocker → insert into nearest minor with reserve room. Otherwise → file with `v1.x` milestone. |
| **Perf gap crossing a published claim** | v1.0 blocker. Next `.x9` hardening minor. |
| **Schema / API surface issue** | **Before v0.84 API-freeze checkpoint:** next minor. **After v0.84:** slips v1.0 unless it's a corruption bug. |
| **Docs / UX polish** | v0.95 final dogfood reserve OR next reserve slot, whichever is closer. |

### Buffer + overflow policy

- **Soft overflow:** when a minor's dogfood reserve fills, the next
  2–3 items roll forward one minor. Planned items keep their slot;
  the *polish scope* of the receiving minor absorbs the overflow.
- **Hard overflow signal:** if reserve overflows by >50% in two
  consecutive minors, the planned-vs-discovery ratio is wrong.
  Pause planned work, drain dogfood first, update this plan.
- **Dogfood beats planned:** if dogfood finds something that
  compromises a v1.0 surface guarantee, it takes priority over a
  FILE-X item from this plan. Some FILE-X items may slip; that's
  the right trade.

### Volume-based axis escalation

If the same axis (e.g., "Bash extractor", "trace direction=both")
generates >3 dogfood issues in a release window, that axis gets a
dedicated hardening slot in the next `.x9`, not patched piecemeal.
Pattern precedent: v0.56 resolver bug family (4 related fixes that
should have been one batched session) — see memory
`project_v0_56_resolver_bug_family.md`.

### Decision authority (so the autonomous loop doesn't over-ask)

| Discovery type | What I do |
|---|---|
| Severity-1 / canonical workflow break | File + ship as patch without asking. |
| Bug in in-flight PR | Fix in same/sibling PR, mention in PR body. |
| Net-new capability surface (new tool / composite / advisory) | File with recommended milestone, then ask before assigning. |
| API-surface change after v0.84 freeze | ALWAYS ask, never assume. |

### Tagging discipline (makes routing auditable)

- `dogfood-found` label on every probe-surfaced issue. Distinct
  from user-reported so we can audit "how much v1.0 work came from
  dogfood vs planning."
- `axis-{ast,doctor,cli,trace,fts,extractor-go,extractor-bash,...}`
  for axis-rotation tracking and the >3-in-a-release escalation
  trigger.
- `severity-{1,2,3}` for the routing table above.

### Capacity reality check for dogfood

Steady-state dogfood surfacing in this repo runs **3–5 PRs/session**
(per memory: v0.69 hit 12 in one big session, v0.71 hit 14 — those
are upper-bound, not mean). 14 minors × 4 avg = **~56 dogfood slots**
absorbed in normal flow. The 84-slot budget elsewhere in this plan
is the aspirational ceiling; the floor is 56. **If we surface >84
net-new items between now and v1.0, something planned slips.**
That's the right outcome — dogfood-found > FILE-X items by default.

### Mandatory release-prep step (adds to CLAUDE.md checklist)

Every release-prep PR adds a "DOGFOOD:" section to the CHANGELOG
entry. Bullet-lists friction found AND fixed in the window. Friction
found but NOT fixed gets filed as an issue with `dogfood-found`
label, never silently dropped. Skipping this section is a release
defect, same severity as skipping the README roadmap-table bump.
